# PRD · LaunchSpec AI

## 1. 产品定位

LaunchSpec AI 是面向小型创业团队和项目负责人的项目方案工作台。它把一句模糊想法转换为可编辑的 MVP 蓝图，并把范围、非目标、验收标准、风险和 AI 人工确认边界显式化，减少“大家以为理解一致”的返工。

- 目标用户：创业团队负责人、外包项目经理、需要快速对齐产品范围的学生项目组。
- 核心场景：项目负责人在开始编码前输入产品想法，生成一份结构化草案；团队编辑、审查后导出为 Markdown。
- 成功指标（MVP）：用户可在 5 分钟内创建、生成、编辑、审查并导出一份不含占位符的方案草案。
- 项目身份：正式 `MSD_DELIVERY_ID` 待课程公布小组号后填写，见 `PROJECT-IDENTITY.md`。

## 2. 问题与机会

产品想法往往分散在聊天记录、口头沟通和临时笔记中。即使有 AI 生成结果，也常是不可测试的长文，缺少范围、验收和风险约束。LaunchSpec AI 将生成内容固定为一个可编辑、可验证的结构，要求人类承担最终判断。

## 3. 用户故事与验收标准

### US-1：创建产品想法

作为项目负责人，我想记录一个清晰的项目名称和产品想法，以便后续生成蓝图。

- 操作：提交不少于 2 个字符的名称和不少于 20 个字符的想法；预期：`POST /api/projects` 返回 `201` 和新项目 ID。
- 操作：提交过短名称、过短想法或非 JSON 请求；预期：接口返回 `400`，且不会创建项目。

### US-2：生成并编辑蓝图

作为项目负责人，我想得到可修改的 MVP 草案，以便让团队先讨论范围而不是直接开工。

- 操作：对已存在项目调用生成；预期：返回目标用户、问题、MVP 范围、非目标、至少两条验收标准、风险、架构和 AI 边界。
- 操作：编辑蓝图并 `PUT /api/projects/{id}` 后重新读取；预期：`GET /api/projects/{id}` 返回已保存的修改内容。

### US-3：审查并导出方案

作为团队成员，我想检查方案是否可交付，并导出一份可放进仓库的 Markdown。

- 操作：对完整蓝图调用审查；预期：返回 `ready` 或 `needs-revision`，并给出范围/验收/非目标/风险/AI 边界相关结论。
- 操作：导出已生成蓝图；预期：`GET /api/projects/{id}/export` 返回 `200`、Markdown 文件和 AI 使用边界。

## 4. MVP 范围

1. 单机本地项目创建与 JSON 持久化。
2. demo provider 与 OpenAI-compatible provider 的结构化蓝图生成。
3. 蓝图编辑、保存、规则/模型审查和 Markdown 导出。
4. `/api/health`、单元测试、检查脚本和本地 API UAT。

## 5. 非目标

- 不自动生成完整业务代码、部署配置或替代产品负责人决策。
- 不做账号体系、多人实时协作、支付、积分、云端对象存储或项目管理工具同步。
- 不把 demo provider 伪装为真实模型能力。
- 不收集真实用户隐私、访问令牌或支付信息。

## 6. 风险与缓解

| 风险 | 缓解 |
| --- | --- |
| 模型输出空泛、格式不正确或超出课程周期 | 固定 JSON 契约、服务端验证、规则审查和人工编辑确认 |
| 真实模型服务不可用或额度不足 | demo 模式保持测试与 UI 可用；真实模型失败如实记录为阻塞项 |
| JSON 文件存储不适合并发 | 明确仅用于本地课程 MVP；后续可迁移至数据库 |
| 团队误把 AI 草案当最终事实 | UI、导出和文档均明确“AI 给草案，人做决策” |
