## 1. 一句话产品说明

arena-lite-2510631109 是一个将同一道题的两个模型回答匿名并列展示、让用户盲投后揭示模型归属并结算 ELO 排行的轻量对战产品。

## 2. 背景与问题

用户在比较模型回答时，容易受模型品牌、既有偏好或展示顺序影响，难以专注判断回答本身的质量；同时，缺少可持续、可解释的对战结果来观察模型在题目上的相对表现。

本阶段以“同题、双回答、匿名盲投、揭盲、ELO 更新”的最小闭环验证：用户能否完成一次基于回答质量的选择，以及系统能否把该选择沉淀为可见的排行榜变化。

## 3. 目标用户与使用场景

### 3.1 目标用户

- 主要用户：希望快速比较两个模型回答质量的课程评审者或体验者。
- 次要用户：需要观察预置模型相对表现的项目维护者。

本阶段用户为预置用户；不提供注册、注册审批、找回密码或账号管理流程。

### 3.2 核心使用场景

1. 预置管理员为 `2510631109-example-question-01` 选择两个不同选手并发起对战，battle 从 `created` 进入 `answering`；双方回答就绪后进入 `ready`。
2. 用户打开 ready 状态的 arena-lite-2510631109 对战，看到一道预置题目及两个匿名回答。
3. 用户仅根据回答内容在“回答 A / 回答 B”中选择更好的一方并提交盲投；系统记录投票并进入 `voted`，投票本身不提前揭盲。
4. 默认投票阈值为 1；达到阈值后，管理员结算对战，battle 进入 `scored`，系统揭示 A、B 对应模型名称与 ELO 变化。
5. 用户查看排行榜，理解预置模型的当前 ELO、名次与本次对战带来的变化。

### 3.3 典型示例（仅示意，不使用真实账号或密钥）

- 题目：`2510631109-example-question-01：解释二分查找的时间复杂度。`
- 模型：`2510631109-model-alpha` 与 `2510631109-model-beta`。
- 预置用户：`2510631109-user-demo`。
- 对战：`2510631109-battle-demo-01`。

## 4. 产品目标与非目标

### 4.1 本阶段目标

- 提供一条从发起对战、匿名比较、盲投、揭盲到 ELO 排行展示的可验收主路径。
- 让用户在投票前无法从界面获知回答 A、B 的模型归属。
- 让一次有效盲投能够对参与模型的 ELO 与排行榜产生可验证的结算结果。
- 保持范围可控：集中期只交付 US-1、US-2 所需的最小体验与证据。

### 4.2 非目标

以下内容明确不属于 PRD v1 或集中期交付范围：

- 注册、登录、注销、找回密码、用户资料编辑、角色权限管理。
- 新建或管理真实模型账号、真实 API Key、Token、支付信息、私有服务地址或在线计费。
- 调用第三方模型服务、模型推理编排、流式输出、模型参数调优或成本计费。
- 多题批量比赛、赛事赛程、锦标赛、模型池自动配对、题库管理后台。
- 平票、弃投、改票、撤销投票、评论、分享、收藏、举报、社交功能。
- 复杂反作弊、真人身份校验、跨设备去重、推荐算法和实时分析看板。
- 移动端专项适配、国际化、无障碍专项优化、生产级监控与高可用部署。
- 任何新的主要业务功能超出 US-1、US-2；现有 `GET /health` 骨架继续保留但不扩展为本 PRD 的实现承诺。

## 6. 用户故事与优先级

### Epic 假设

我们相信：对预置用户将同题的两个模型回答匿名并列呈现，并在盲投达到默认阈值、完成一次合法结算后揭盲和显示 ELO 变化，能够让用户以回答质量而非模型品牌完成比较；通过盲投完成率、揭盲可见性和排行榜结算证据验证该闭环。

### US-1：发起对战并盲投（P0，必须）

**用户故事**

作为预置用户 `2510631109-user-demo`，我想查看 `2510631109-battle-demo-01` 中同一道题的两个匿名回答并选择更好的一方，以便在不知道模型归属的情况下完成公平比较。

**验收标准**

1. 给定有效的 `2510631109-battle-demo-01`，当用户进入对战页时，页面显示题目、回答 A、回答 B 与可选择的投票操作；A/B 的模型真实名称、ELO、名次和其他可识别归属的信息在投票前均不展示。
2. 当用户选择回答 A 或回答 B 并提交时，系统仅接受 A/B 两种有效选择，记录该预置用户对该对战的一次盲投，并使 battle 从 `ready` 进入 `voted`；投票响应不提前暴露 A/B 对应模型。
3. 当对战不存在、选择值不是 A/B，或对战缺少同题的两份可展示回答时，系统不结算 ELO，并给出明确失败反馈。
4. 当同一预置用户对同一 `2510631109-battle-demo-01` 再次提交投票时，系统拒绝重复提交，不重复结算，且明确说明该对战已投票。

**优先级理由**

没有匿名呈现和有效盲投，就没有可信的对战输入，US-2 无法成立。

### US-2：揭盲与排行榜（P0，必须）

**用户故事**

作为完成盲投并已被结算的预置用户 `2510631109-user-demo`，我想看到回答 A/B 对应模型、获胜结果、ELO 变化和排行榜，以便理解我的选择如何影响模型的相对排名。

**验收标准**

1. 在 US-1 的有效投票达到阈值、且管理员完成一次合法结算后，系统揭示回答 A/B 对应的 `2510631109-model-alpha`、`2510631109-model-beta`（或同样符合命名约束的预置模型）、用户选中的获胜回答，以及双方投票前后 ELO 或本次 ELO 变化。
2. 系统按已定义且一致的 ELO 规则，只对本场参与的两个模型结算一次；胜方 ELO 增加、负方 ELO 减少，且双方变化量代数和为 0。
3. 用户可查看排行榜；排行榜按当前 ELO 从高到低排列，并显示模型名称、ELO 和名次。若 ELO 相同，使用预先声明的稳定次级排序规则。
4. 如果对战尚未完成有效投票、模型数据缺失，或结算前置条件不满足，系统不展示伪造的结算成功信息，也不得修改排行榜。

**优先级理由**

揭盲与排行榜把盲投转换为可解释的系统反馈，是验证 ELO 对战价值的闭环终点。

## 7. 核心用户流程

1. admin 发起 `2510631109-battle-demo-01`，系统从 `created` 并发准备两份回答，成功后进入 `ready`；失败则进入 `aborted` 并保留原因。
2. 用户在 ready 状态看到题目与两个匿名回答 A/B；系统不泄露模型归属、ELO 或名次。
3. 用户选择 A 或 B 并提交。
4. 系统校验对战状态、投票选择、预置用户状态和是否重复投票。
5. 校验成功后，系统保存一次投票，状态进入 `voted`，不提前结算或揭盲。
6. 达到默认阈值 1 后，admin 结算一次；状态进入 `scored`，系统揭示模型归属、胜负结果、ELO 变化，并展示更新后的排行榜。
7. 校验或结算失败时，系统保持原有 ELO/排行榜状态并展示错误反馈；不进入成功揭盲或 scored 状态。

## 8. 关键产品规则

### 8.1 匿名与揭盲规则

- 匿名阶段只使用“回答 A”“回答 B”标记；不得通过模型名称、ELO、名次、头像、链接、样式差异或隐藏文本泄露归属。
- 揭盲的触发条件是有效盲投达到默认阈值、且 admin 对该对战的一次合法结算已完成。
- 投票前的匿名状态与投票后的揭盲状态必须可区分；不得在投票前提供绕过揭盲的入口。

### 8.2 投票规则

- 单个对战由同一道题的恰好两份回答组成。
- 有效选项仅为 A 或 B；本阶段不支持平票、跳过、弃投或多选。
- 一个预置用户对一个对战仅能产生一次有效投票；重复尝试不改变结果或 ELO。

### 8.3 ELO 与排行榜规则

- 本阶段采用标准双人零和 ELO 更新：
  - 期望胜率：`E_A = 1 / (1 + 10 ^ ((R_B - R_A) / 400))`。
  - 用户投 A 时：`R_A' = R_A + K × (1 - E_A)`，`R_B' = R_B + K × (0 - (1 - E_A))`。
  - 用户投 B 时对称计算。
- PRD v1 的默认假设为固定 `K = 32`；实现前可在 SPEC/ADR 中确认，但一经选定必须对所有本阶段结算一致执行并可测试。
- 排行榜按当前 ELO 降序排列；相同 ELO 时按预置模型标识的字典序升序作为稳定次级排序，不以随机顺序决定名次。
- ELO 结算只发生在一次合法 settle 使 battle 从 `voted` 进入 `scored` 时；失败、无效或重复投票、重复 settle 均不得改变分数。

## 10. 关键风险、假设和依赖

### 10.1 关键风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 匿名信息泄露 | 用户受品牌偏好影响，盲投失效 | 对匿名状态单独测试；审查名称、ELO、名次、隐藏文本和 UI 差异。 |
| 重复投票或重复结算 | ELO 被错误放大，排行榜失真 | 为“预置用户 + 对战”设置一次性投票约束；重复路径不修改状态。 |
| ELO 公式或排序实现不一致 | 结果无法解释或无法复现 | 在 SPEC/ADR 固化 K 值、计算顺序、四舍五入与同分排序；用确定性测试覆盖。 |
| 回答质量差异过小或题目有歧义 | 单次投票的信号弱 | 本阶段只验证产品闭环，不宣称排行榜代表普遍模型能力；后续扩展题库和投票样本。 |
| 范围蔓延 | 集中期无法完成可验收闭环 | 所有新增需求先映射到 US-1/US-2；不能映射的内容进入后续候选。 |
| 预置数据不完整 | 无法发起有效对战或结算 | 在实现与测试前确认每个 `2510631109-battle-demo-01` 均有题目、两个模型、两份回答和初始 ELO。 |

### 10.2 假设

- 本阶段可使用脱敏、预置的用户、题目、模型、回答和初始 ELO 数据，不需要真实账号或外部模型服务。
- 每场对战只包含两个不同的预置模型，且两份回答针对同一道题。
- 用户选择 A 或 B 可作为该场对战的胜负信号；本阶段不做统计显著性推断。
- 单进程、单课程演示场景足以验证流程；不以并发、高可用或生产规模为验收目标。
- 当前 `GET /health` 最小骨架是已有能力；业务实现必须在已接受任务卡与后续 SPEC 后另行进行。

### 10.3 依赖

- 已有 FastAPI 项目骨架与通过的 `/health` 测试。
- 后续 `docs/SPEC.md` 对数据结构、状态、错误路径、ELO 取整/存储细节和接口契约作出可测试约定。
- 预置的脱敏演示数据，例如 `2510631109-user-demo`、`2510631109-battle-demo-01`、`2510631109-model-alpha`、`2510631109-model-beta`。
- 开发、QA 与 reviewer-2510631109 对范围、错误路径、测试命令和证据的复核；不依赖第三方账号、密钥、付费额度或私有服务。

## 11. 开放问题与下一步验证

1. ELO 在展示与持久化时采用整数、固定小数还是内部高精度计算后展示？建议在 SPEC/ADR 中固定，并配套边界测试。
2. 初始 ELO 是否统一设定为一个课程演示默认值，还是允许每个预置模型携带不同初值？建议本阶段先统一默认值以便解释。
3. 匿名页面的审查范围是否包含原始响应或仅包含 UI？建议两者均验证，避免客户端数据泄露模型归属。
4. 重复投票的用户体验采用“拒绝并显示已投票”还是“返回已揭盲结果”？建议在 SPEC 中选择其一，并确保不会再次结算。
